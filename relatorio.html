Observações Técnicas

Diferença entre Fork e Clone: O Fork cria uma cópia remota do repositório no GitHub vinculada ao original, permitindo contribuir com Pull Requests. O Clone copia o repositório para a máquina local para trabalho offline.

3. Git Flow (Branches)

Branches Criadas

master develop feature/health-endpoint feature/improve-health-endpoint release/1.0.0

Comandos Executados

# Criar branch develop
git checkout -b develop
git push -u origin develop

# (Opcional) Inicializar git-flow
git flow init
# production branch: master
# development branch: develop
# feature prefix: feature/
# release prefix: release/
# hotfix prefix: hotfix/
# support prefix: support/
# version tag prefix: v

Comentários

A branch develop serve como integração para todas as features antes de ir para produção. Isso permite que o código em master (produção) permaneça estável enquanto novas funcionalidades são testadas e revisadas em develop.

Fluxo do Colega

git pull

# Colega cria branch de melhoria
git checkout -b feature/improve-health-endpoint

# Implementa melhoria: versão configurável via appsettings.json
# Commit e push
git add .
git commit -m "feat: health endpoint now returns app version from configuration"
git push -u origin feature/improve-health-endpoint

Melhoria Implementada

O endpoint /api/health foi refatorado para ler a versão e nome do aplicativo diretamente do appsettings.json usando IConfiguration, removendo valores hardcoded.

Antes:

Version = "1.0.0",
Application = "TodoApi"

Depois:

Version = config["Application:Version"] ?? "1.0.0",
Application = config["Application:Name"] ?? "TodoApi"

Code Review e Merge

  • Review: ✅ LGTM! Aprovada com elogio à melhoria de configuração externa
  • Merge: Merge via Squash no GitHub → develop
  • Linked Issue: Closes #1

7. Release e Tag

Branch de Release

git checkout -b release/1.0.0
# Ajustes finais de versão e changelog
git commit -m "chore: bump version to 1.0.0"
git push origin release/1.0.0

Merge na Master e Criação da Tag

git checkout master
git merge develop --no-edit

# Criar tag anotada
git tag -a v1.0.0 -m "Initial release - Health Endpoint v1.0.0"
git push origin v1.0.0

9. Reflexões Finais

Aprendizados

  • Compreensão prática do Git Flow com branches main, develop e feature/*
  • Experiência real com Pull Requests, code review e merge workflow
  • Importância de mensagens de commit padronizadas (Conventional Commits: feat:, chore:, docs:)
  • Diferença entre fork, clone e branch
  • Uso de tags anotadas para releases (git tag -a v1.0.0)
  • Vinculação de Issues com PRs usando Closes #n

Dificuldades e Soluções

DificuldadeSolução
Entender quando usar merge vs rebasePesquisa na documentação do GitHub; merge para preservar histórico, rebase para linearizar
Configurar acesso SSH ao GitHubGerar SSH key com ssh-keygen e adicionar em Settings → SSH Keys
Resolver conflitos de mergeUsar git status para identificar arquivos em conflito e editar manualmente
Padronizar mensagens de commitAdotar Conventional Commits: feat:, fix:, docs:, chore:

Boas Práticas que Adotarei

  • Sempre criar branch de feature a partir de develop, nunca de main
  • Usar mensagens de commit descritivas e padronizadas
  • Sempre descrever PRs com contexto, motivação e instruções de teste
  • Fazer code review cuidadoso antes de aprovar merges
  • Usar tags anotadas para releases com descrições significativas
  • Vincular Issues a PRs para rastreabilidade
  • Manter .gitignore atualizado para evitar commit de arquivos sensíveis
